# Safety Incident Playbook Draft

Last updated: 2026-04-22
Status: internal operations draft

## 1. Purpose

This playbook is for responding to safety-sensitive incidents on Neonadri.

## 2. Trigger categories

Escalate when any of the following appears:

- threat, coercion, or stalking language
- payment for attendance, time, or companionship
- sexual or romantic quid pro quo language
- suspected underage user access
- serious complaint tied to an in-person meetup
- repeat offender patterns across posts or chats

## 3. Immediate first response

When a high-risk issue is identified:

1. capture the relevant account, meetup, match, chat, and report context
2. assess whether the risk appears immediate, active, or ongoing
3. decide whether content or account access should be temporarily restricted
4. document the reason for the restriction and the evidence reviewed

## 4. Evidence to preserve

Preserve at least:

- user IDs
- meetup ID and post text
- related chat context
- report text
- timestamps
- prior enforcement history if relevant

## 5. Recommended action levels

### Level 1: low or unclear risk

Examples:

- ambiguous language
- first-time minor policy issue

Action:

- review
- document
- warn or monitor as needed

### Level 2: moderate safety concern

Examples:

- repeated unsafe payment language
- aggressive conduct
- evasion after a block

Action:

- restrict content
- temporarily limit account features
- escalate to admin review

### Level 3: urgent or severe concern

Examples:

- credible threat
- coercive payment proposal
- suspected minor in adult service
- serious post-meetup safety complaint

Action:

- disable relevant content or account access immediately
- preserve records
- escalate for senior review without delay

## 6. Follow-up checklist

- confirm what action was taken
- record whether the user appealed or contacted support
- identify whether product rules or UI should be tightened
- note whether the event suggests a broader pattern

## 7. Release-readiness note

Before launch, this playbook should be aligned with actual admin tools, report inputs, retention behavior, and any legal escalation path.
